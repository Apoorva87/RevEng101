Authorization Code Interception

The specific attack PKCE was invented to stop — and proof that it works.

The Core Problem

OAuth 2.0's authorization code flow has an inherent weakness: the authorization code travels through the user's device on its way from the auth server back to your app. It rides in a URL — either a custom URI scheme redirect or an HTTP redirect to localhost. That URL passes through layers of software the app developer doesn't control: the OS URL dispatcher, the browser, browser extensions, other apps registered for the same URI, OS accessibility services, clipboard managers, and more.

If any of those layers is compromised or misconfigured, an attacker can grab the code. And without PKCE, that code is all they need to get tokens.

Analogy: The Postcard Problem

Think of the authorization code as a postcard being mailed to your house. The message (code) is written in the open — anyone who handles it along the way can read it. A letter (HTTPS to the token endpoint) is sealed — only the recipient can open it. The problem is: OAuth must use a postcard for one hop (the redirect through the browser), even though every other hop uses sealed letters.

PKCE says: fine, the postcard is readable — but make it useless without a secret only the sender knows.

The Anatomy of the Redirect

Here's what happens at the critical moment — when the auth server sends the code back:

Real Attack Techniques

These aren't theoretical. Each technique has been demonstrated in the wild or in published security research.

1 Custom URI Scheme Hijacking (Mobile)

Platform: Android, iOS (pre-Universal Links)

How it works: On mobile, OAuth redirects typically use custom URI schemes like com.myapp://callback. On Android, any app can register an intent filter for any custom scheme — and the OS doesn't guarantee which app receives the redirect. A malicious app declares the same scheme in its manifest:

Real-world impact: This was so common that RFC 7636 (PKCE, published 2015) was written specifically because of it. The RFC abstract says: "...a malicious application could intercept the authorization code returned from the authorization endpoint..."

With PKCE: The evil app gets the code, but can't exchange it — it doesn't have the code_verifier that only existed in the real app's memory.

2 Localhost Port Race (Desktop)

Platform: Windows, macOS, Linux

How it works: Desktop OAuth apps (like Codex CLI, gcloud auth login, Spotify desktop) start a temporary HTTP server on a localhost port to catch the redirect. An attacker who can run code on the same machine can race to bind that port first.

Variations:

  • SO_REUSEPORT abuse: On Linux, any process can set SO_REUSEPORT and bind the same port. The kernel load-balances between them — the attacker gets ~50% of connections.
  • Process crash timing: If the legitimate app crashes between starting the auth flow and receiving the callback, the port is free for the taking.
  • Predictable ports: OpenAI's Codex uses port 1455 always. An attacker who knows this can pre-bind the port.

With PKCE: The attacker catches the code but the code_verifier died with the legitimate process (or never existed in the attacker's memory).

3 Browser Extension Interception

Platform: All desktop platforms

How it works: Browser extensions with the webRequest or webNavigation permission can observe every URL the browser navigates to — including OAuth redirects. A malicious or compromised extension silently reads the authorization code from the redirect URL.

Scale: Compromised browser extensions have affected millions of users. In 2020, a campaign involving 28 malicious extensions (with 3M+ installs) was discovered redirecting users and stealing browsing data. Extensions like "The Great Suspender" (2M users) were found to be compromised after being sold to new owners.

With PKCE: The extension can read the URL (can't prevent that), but the stolen code is useless without the code_verifier stored in the app's process memory — which the extension cannot access.

4 Inter-App Communication Leaks (iOS)

Platform: iOS (pre-ASWebAuthenticationSession)

How it works: Before iOS 11, apps used SFSafariViewController or an external Safari browser for OAuth. The redirect back to the app used a custom URI scheme. iOS didn't have exclusive URI scheme ownership — multiple apps could register the same scheme, and the behavior was undefined (usually last-installed wins).

Apple's fix: ASWebAuthenticationSession (iOS 12+) returns the redirect directly to the calling app via the OS, bypassing URI scheme dispatch entirely. But PKCE remains essential because older iOS versions and cross-platform apps still use custom schemes.

5 Log File / History Leaks

Platform: All

How it works: The authorization code appears in places that persist beyond the moment of the redirect:

  • Browser history: http://localhost:1455/auth/callback?code=a1b2c3 is saved in browsing history
  • HTTP server logs: The callback server may log the request (access.log)
  • Proxy logs: Corporate HTTP proxies log all URLs, including localhost redirects if the browser goes through the proxy
  • Referrer header: If the callback page loads any external resource, the Referer header could leak the full callback URL (including the code) to that external server
  • Shell history: If you manually run a curl command with the code, it appears in ~/.bash_history

With PKCE: Even if the code is found in logs weeks later, it's expired and useless without the code_verifier — double protection. Without PKCE, a code found in logs is potentially replayable (if not yet exchanged and not yet expired).

6 OS Accessibility & Clipboard Sniffing

Platform: Windows, macOS

How it works: Assistive technology APIs (like UIAutomation on Windows or Accessibility APIs on macOS) can read the content of any window — including the browser's address bar. Malware with accessibility permissions can watch the URL bar and extract the authorization code the moment the redirect occurs.

On some platforms, the URL may also briefly appear in clipboard history if the browser auto-copies it, or if the user copies it for debugging.

With PKCE: The URL is readable, but the code alone is worthless.

Why Can't the Attacker Also Steal the code_verifier?

This is the key insight. The code_verifier is a random string that:

Property What it means
Lives only in process memory Never written to disk, never in a URL, never in a header, never in a log. It's a variable in your app's RAM.
Sent only over HTTPS Goes from your app → auth server in a POST body over TLS. Only the auth server and your app ever see it.
Never touches the browser The browser only sees the code_challenge (SHA-256 hash). Even a fully compromised browser can't reverse the hash.
Never in a redirect URL Redirect URLs are logged, cached, and visible to extensions. The verifier travels through none of these.
The attacker's dead end

The attacker has the code and the code_challenge (the hash). To exchange the code for tokens, they need the code_verifier. To get the verifier from the challenge, they'd need to reverse SHA-256 — which is computationally impossible. The code is useless. PKCE wins.

The Full Attack vs. Defense Timeline

Historical Context: Why PKCE Was Created

2012 — OAuth 2.0 (RFC 6749)
OAuth 2.0 published. Public clients (mobile/desktop apps) use the authorization code flow but have no client_secret — anyone can impersonate the client. The redirect is the only thing linking the code to the app.
2013-2014 — Mobile OAuth attacks proliferate
Researchers demonstrate custom URI scheme hijacking on Android and iOS. Banking apps, social media apps, and enterprise SSO apps are all vulnerable. The attack is trivial: install an app that registers the same URI scheme.
2015 — RFC 7636: PKCE published
IETF publishes "Proof Key for Code Exchange by OAuth Public Clients." Originally recommended only for public clients (mobile/desktop). Key insight: bind the authorization request to the token exchange using a one-time proof that only the legitimate app possesses.
2017 — OAuth 2.0 for Native Apps (RFC 8252)
IETF recommends PKCE for ALL native app OAuth flows. Also recommends using the system browser (not embedded WebViews) and claimed HTTPS redirect URIs (Universal Links on iOS, App Links on Android) instead of custom URI schemes.
2020 — OAuth 2.0 Security BCP (RFC 6819 update)
IETF draft recommends PKCE for all OAuth clients — including confidential (server-side) ones. PKCE protects against code injection attacks even when a client_secret exists.
2023-2025 — Universal adoption
Major providers (Google, Microsoft, Apple, GitHub, OpenAI) all require or strongly encourage PKCE. OpenAI's Codex CLI uses it for desktop CLI OAuth. OAuth 2.1 (draft) makes PKCE mandatory for all flows.

In the OpenAI Codex Flow Specifically

Here's why PKCE matters in the exact flow our scripts/auth/setup.sh uses:

Attack Vector Applicable? PKCE Protects? Details
Custom URI scheme hijack No — We use http://localhost:1455, not a custom scheme
Localhost port race Yes Yes ✓ Port 1455 is hardcoded and predictable. Another process could race to bind it.
Browser extension sniffing Yes Yes ✓ Any extension with webRequest permission sees the callback URL.
Browser history / log leak Yes Yes ✓ The callback URL with the code is saved in browser history.
Referrer header leak Low Yes ✓ Our callback page is plain HTML with no external resources, but if it did...
Full network MITM Maybe No ✗ If attacker controls the network AND can intercept HTTPS → they see everything including verifier. But this requires breaking TLS.
Same-machine malware Maybe Partial Malware with process memory access could read the verifier. But malware that can only intercept URLs/ports is stopped by PKCE.
Bottom Line

PKCE doesn't claim to protect against an attacker who owns your machine or breaks TLS. It protects against the realistic, common case: something on your machine can see the redirect URL, but can't read another process's memory or break HTTPS encryption. That's browser extensions, competing apps, log scrapers, port racers, and clipboard watchers. This covers the vast majority of real-world OAuth attacks on native apps.
