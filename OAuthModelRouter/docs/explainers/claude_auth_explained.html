Claude OAuth Flow: How It Actually Works

Reverse-engineered from Claude Code CLI v2.1.105 (Bun binary) and confirmed against live endpoints. April 2026.

The Concert Wristband Analogy

Getting into a concert venue works like this:

  1. You show your ticket at the gate (browser login at claude.com)
  2. They give you a wristband valid for tonight's show (access_token, ~8 hours)
  3. You also get a stamp on your hand (refresh_token) to get back in if you step out
  4. The stamp is single-use — each re-entry replaces it with a new stamp

PKCE is like the bouncer asking for the other half of a torn ticket — only the person who originally requested entry has the matching half.

3
User logs in at Claude (email, Google SSO, etc.). Claude verifies the user's identity.
4

5
Exchange the code for tokens via POST to platform.claude.com/v1/oauth/token. The body is JSON (not form-encoded!) with the code, code_verifier, client_id, and state. Claude hashes the verifier, checks the match, and returns tokens.
6
Use the tokens! The access_token (valid ~8 hours) goes in Authorization: Bearer header. Critical: you must also send anthropic-beta: oauth-2025-04-20 or the API rejects OAuth tokens. The response also includes account.email_address and organization info directly.

Token Refresh

IP-Based Rate Limiting on Token Refresh

The platform.claude.com/v1/oauth/token endpoint rate-limits by IP address, not per-account. If you have multiple Claude accounts on one machine (via CLAUDE_CONFIG_DIR isolation), a failed refresh for one account locks out ALL accounts from that IP. This was the primary motivation for switching to long-lived API keys in OAuthModelRouter.

The Cloudflare Trap We Hit

How We Discovered This

We spent hours debugging persistent 429 errors. The breakthrough came from intercepting a successful Claude Code login and comparing headers. The only difference was User-Agent:

  • Our script: User-Agent: claude-code/2.1.105 → 429 every time
  • Claude Code (via axios): User-Agent: axios/1.13.6 → 200 OK

The binary only sets Content-Type: application/json for token requests. It does NOT override User-Agent — axios fills in its default. The claude-code/2.1.105 UA is used for API calls, not OAuth token exchange.

What Is Axios and Why Does It Matter Here?

Think of it like mailing a letter

Your code writes the letter (the JSON body with grant_type, code, etc.). But it needs an envelope and a postal service to deliver it. Axios is the postal service — it handles the actual HTTP transport: opening the TCP connection, setting default headers, managing timeouts, following redirects, and parsing the response. When you hand axios a letter, it stamps its own return address on the envelope (User-Agent: axios/1.13.6) unless you explicitly tell it not to.

?
What is axios?
Axios is a popular JavaScript/TypeScript HTTP client library (50M+ weekly npm downloads). It wraps the lower-level http/https modules in Node.js (or XMLHttpRequest in browsers) and provides a cleaner API: automatic JSON serialization, request/response interceptors, timeout handling, and sensible defaults. Claude Code’s binary is compiled with Bun, but it still bundles axios as its HTTP client for token operations.

Why Claude Code Uses Axios (Not Fetch)

Modern JavaScript has a built-in fetch() API, and Bun natively supports it. So why does Claude Code bundle a third-party library? A few reasons:

fetch()
  • Built-in, zero dependencies
  • No default User-Agent
  • Returns a Response object — must call .json() manually
  • No timeout support (need AbortController)
  • No request/response interceptors
  • Does not throw on 4xx/5xx — must check response.ok
axios
  • ~50 KB bundled, mature ecosystem
  • Sets User-Agent: axios/1.13.6 by default
  • Auto-serializes JS objects to JSON
  • Built-in timeout option (Claude Code uses 15s)
  • Interceptors for auth headers, logging, retries
  • Throws on 4xx/5xx — natural try/catch error handling

The Accidental Security Feature

Axios’s default User-Agent accidentally bypasses Cloudflare rate limiting

Here’s the chain of events that makes this work:

  1. Claude Code’s token exchange calls f6.post(url, body, {headers: {"Content-Type": "application/json"}, timeout: 15000})
  2. Notice: no User-Agent in the explicit headers
  3. Axios fills in its default: User-Agent: axios/1.13.6
  4. Cloudflare sees a known HTTP client library UA and lets it through
  5. If you set User-Agent: claude-code/2.1.105 (or any custom string), Cloudflare treats it as a bot/scraper and aggressively rate-limits with 429

The claude-code/2.1.105 UA (built by the K9H() function in the binary) is used for API calls to api.anthropic.com, which has different Cloudflare rules. The OAuth token endpoint at platform.claude.com is stricter. Because the Claude Code developers used axios and didn’t bother overriding the User-Agent for token calls, the default axios UA became the “correct” one.

What This Means for Your Scripts

If you’re writing your own tool that needs to exchange tokens with platform.claude.com, you must set the User-Agent to a known HTTP client library. The safest options:

The takeaway

Axios isn’t just an HTTP convenience wrapper here — its default User-Agent is load-bearing. Without it, the entire OAuth flow breaks. This is almost certainly unintentional on Anthropic’s part: the Cloudflare rules were likely set up to block scrapers and bots, and nobody noticed that the rule also catches Claude Code’s own branded UA because the code never sends it to that endpoint. The lesson: when reverse-engineering auth flows, don’t just capture the body — capture every header, especially the ones set by the HTTP library, not the application.

Claude vs. OpenAI: Side-by-Side Comparison

The Surprising Differences

Same Protocol (OAuth 2.0 + PKCE), Very Different Implementations

Both flows follow RFC 6749 and RFC 7636, but diverge in almost every detail:

OpenAI: Form-Encoded Exchange

The token exchange uses application/x-www-form-urlencoded — standard per the OAuth spec. This is what most OAuth implementations use.

But refresh uses JSON! Two different content types for the same endpoint.

Claude: JSON Everything

Both code exchange AND refresh use application/json. Non-standard but consistent. This is critical — sending form-encoded to Claude's token endpoint triggers Cloudflare rate limiting.

OpenAI: Fixed Port 1455

The redirect URI localhost:1455 is hardcoded in OpenAI's OAuth client registration. Any other port returns unknown_error. If port 1455 is occupied, the login fails.

Claude: Dynamic Port

Claude accepts any localhost port in the redirect_uri. The script picks a port, binds to it, and sends that in the authorize URL. More flexible — no port conflicts possible.

OpenAI: JWT Access Tokens

The access_token is a JWT you can decode to find email, plan type, and account ID. Lasts ~10 days. Contains https://api.openai.com/auth claims.

Claude: Opaque Tokens + Metadata

The access_token (sk-ant-oat01-...) is opaque — you can't decode it. Lasts only ~8 hours. But the token response includes account and organization objects with email and org info.

OpenAI: No Cloudflare on Token Endpoint

auth.openai.com/oauth/token has zero Cloudflare protection. Any curl command works. The trap is that OIDC discovery points to auth0.openai.com which IS behind Cloudflare.

Claude: Cloudflare + UA Filtering

platform.claude.com is behind Cloudflare. It accepts known HTTP client UAs like axios/1.13.6 but rate-limits custom UAs. Claude Code's binary only sets Content-Type — axios provides its own default UA.

Claude-Specific Gotchas

The code=true Parameter

Claude's authorize URL requires code=true as a query parameter. Without it, the auth page may not return an authorization code. This is not standard OAuth — it's Claude-specific. It tells the Claude auth SPA to use the authorization code flow rather than an implicit or session-based flow.

The anthropic-beta Header

The Anthropic API rejects OAuth tokens unless you include anthropic-beta: oauth-2025-04-20 in every API request. Without it, you get: "OAuth authentication is currently not supported". API keys don't need this header. The router automatically injects it for all Claude proxy requests.

Scope Filtering: org:create_api_key

You can request org:create_api_key in the authorize URL, but the auth server does not grant it. The granted scopes are: user:profile user:inference user:sessions:claude_code user:mcp_servers user:file_upload. This means you cannot programmatically create long-lived API keys through this flow.
