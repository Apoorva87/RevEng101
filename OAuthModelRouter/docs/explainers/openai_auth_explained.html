OpenAI OAuth Flow: How It Actually Works

Reverse-engineered from the Codex CLI (Rust source) and confirmed against live endpoints. April 2026.

The Hotel Key Card Analogy

Imagine checking into a hotel. You don't get a copy of the master key — instead:

  1. You show your ID at the front desk (browser login at OpenAI)
  2. They give you a key card that only opens your room (access_token)
  3. The card expires in 10 days, but you also get a voucher (refresh_token) to get a new one
  4. The voucher is single-use — each time you use it, they give you a new card AND a new voucher

PKCE is like the front desk asking you a secret question that only you set up — it proves the person picking up the key card is the same person who asked for the room.

The Big Picture

Step by Step

1
Generate a PKCE pair — a random code_verifier (secret, stays with you) and its SHA-256 hash called code_challenge (sent to OpenAI). This proves the same app that started the login is the one finishing it.

SHA-256 →
code_challenge (public)

2
Open browser to OpenAI's authorize URL. The URL includes the code_challenge, your client_id, requested scopes, and a random state value (CSRF protection). The user sees the OpenAI login page.
3
User logs in at OpenAI (email, Google, etc.). OpenAI verifies the user's identity.
4
OpenAI redirects the browser to http://localhost:1455/auth/callback?code=abc123&state=xyz. Your local server (running on port 1455) captures this code. The code alone is useless without the code_verifier — even if someone intercepts it, they can't use it.
5
Exchange the code for real tokens. Your script sends a POST to auth.openai.com/oauth/token with the code AND the original code_verifier. OpenAI hashes the verifier, checks it matches the challenge from step 2, and returns the tokens.
6
Use the tokens! The access_token (a JWT valid for ~10 days) goes in the Authorization: Bearer header for API calls. The refresh_token gets you new tokens when the access_token expires.

The Cloudflare Wall We Hit

The Trap: OIDC Discovery Lies

OpenAI's .well-known/openid-configuration says the token endpoint is auth0.openai.com/oauth/token. But that domain is behind Cloudflare and returns 403 for ALL programmatic clients (curl, Python, Node.js, even TLS-impersonating libraries). The actual endpoint that works is auth.openai.com/oauth/token — confirmed by reading the Codex CLI Rust source code.

Can MITM Defeat This Flow? (Yes.)

You're right to question this. If an attacker has a full Man-in-the-Middle position — meaning they can read AND modify all traffic between your machine and OpenAI — then PKCE does NOT save you. Here's why:

So What Does PKCE Actually Protect Against?

PKCE was designed for a very specific threat: authorization code interception. This is when an attacker can steal the code from the redirect, but cannot see the direct HTTPS traffic between your app and the token endpoint. [Deep dive: 6 real attack techniques →]

The Real Threat Model: Mobile Apps

PKCE was invented for mobile apps (RFC 7636, 2015). On a phone, when the browser redirects to myapp://callback?code=abc, a different malicious app could register the same custom URL scheme and steal the code. The malicious app gets the code but never saw the code_verifier (which lived only in the legitimate app's memory). PKCE stops this.

On desktop, the equivalent is: another process on your machine could try to bind port 1455 before your script does, or intercept the localhost redirect via a browser extension. PKCE protects against that.

The key distinction: the attacker intercepts the redirect (the code traveling via the browser), but cannot see the direct HTTPS POST from your app to the token endpoint. The verifier never travels through the browser — it goes straight from your app's memory to OpenAI over HTTPS.

The Honest Truth About This Flow

Our scripts/auth/setup.sh uses HTTP (not HTTPS) for the localhost callback (http://localhost:1455). The code travels unencrypted on the loopback interface. This is fine because:

  • Only processes on YOUR machine can see loopback traffic
  • If an attacker has code execution on your machine, you have bigger problems
  • The token exchange (code-for-tokens) goes over HTTPS to auth.openai.com

But: the code_verifier is embedded in the Python process via bash variable interpolation. Any process that can read /proc/[pid]/cmdline or ps aux could theoretically see it. For a local dev tool, this is an acceptable risk. For a production system, you'd generate the verifier inside the process that uses it.

Client ID, Scopes, and State: How They Work

Client ID

What is it?

A client_id identifies which application is requesting access. It's like the name badge a delivery driver wears — it tells the building security who they are, but it's NOT a secret (anyone can read the badge).

AspectIn GeneralIn This Flow
Who creates it? You register your app with the OAuth provider and get one OpenAI created it for the Codex CLI. We reuse it: app_EMoamEEZ73f0CkXaXp7hrann
Is it secret? No. Public clients (mobile/CLI apps) cannot keep secrets. It's in the URL. No. It's visible in the authorize URL and the Codex binary. That's fine.
What's tied to it? Allowed redirect URIs, scopes, rate limits, branding Only localhost:1455 is registered. Other ports are rejected by OpenAI.
Can anyone use it? Yes, but tokens are scoped to the user who logs in, not the client_id Yes. We use the same client_id as Codex. The tokens belong to whoever logs in.

Scopes

What are they?

Scopes are permission requests. Like an employee badge that gives access to certain floors — you ask for the floors you need, and the building grants or denies each one.

ScopeWhat It MeansWhy We Need It
openid Return an id_token proving the user's identity Required by the OIDC protocol. Gets us the JWT with email/account info.
profile Include the user's name in the token Nice to have for display purposes
email Include the user's email in the token We use it to label the token in the DB (e.g., "codex-plus : user@gmail.com")
offline_access Return a refresh_token Without this, we'd have to re-login every 10 days when the access_token expires
api.connectors.read Read Codex connectors Codex-specific API access. Required for the token to work with Codex endpoints.
api.connectors.invoke Invoke Codex connectors Allows making API calls (chat completions, etc.)

How scopes are chosen in general: You request the minimum permissions your app needs. The OAuth provider may show a consent screen listing them. The user can sometimes decline individual scopes. Over-requesting scopes makes users suspicious and may get your app rejected during review.

In this case: We copy the exact scopes from the Codex CLI. If we requested fewer (e.g., dropped api.connectors.invoke), the access_token wouldn't have permission to make API calls. If we requested more (e.g., added scopes that don't exist), the auth server would reject the request.

State

What is it?

The state parameter is a CSRF token — a random value you generate, attach to the authorize request, and verify when it comes back. It's like putting a sticky note on your desk that says "I'm expecting a delivery with reference #X7k9" — if someone shows up with a different reference number, you know it's not the delivery you ordered.

How state is chosen: It should be a cryptographically random value (we use 32 bytes, base64-encoded). Your app generates it, remembers it, sends it with the authorize request, and checks that the callback has the same value. If it doesn't match, someone else initiated the flow.

Note: Our scripts/auth/setup.sh currently generates state but doesn't validate it on the callback — it relies on the fact that only one login can be in progress at a time on port 1455. A production implementation should verify the state matches.
