Dashboard walkthrough and endpoint map

This page explains how the current DB-backed dashboard works, how the browser code in src/oauthrouter/static/portal.html and src/oauthrouter/static/portal.js talks to the FastAPI app assembled in src/oauthrouter/server.py, and what every HTTP endpoint is responsible for.

How the dashboard boots

The home screen is intentionally thin: it fetches DB state, renders views, and posts mutations back to the API.

1. Initial load

DOMContentLoaded → loadTokens() is the entrypoint. That function requests /api/tokens, /health, and /api/providers in parallel.

2. Render pass

The response fan-out is:

  • renderStats() for the top summary cards
  • renderTokens() for the token table
  • renderProviders() for provider controls and test state
  • setEndpointPorts() to rewrite the copyable command snippets

3. State kept in the browser

The dashboard keeps only lightweight UI state in memory:

  • providerCache for the last provider payload
  • providerTestState for the latest provider test result card
  • providerSelections for the currently selected token per provider
  • editingProviderName and token edit modal state

4. Token actions

Table actions call simple mutation endpoints:

  • addManualToken() → POST /api/tokens
  • updateTokenPriority() → PATCH /api/tokens/{id}
  • toggleToken() → POST /api/tokens/{id}/enable|disable
  • refreshToken() → POST /api/tokens/{id}/refresh
  • deleteToken() → DELETE /api/tokens/{id}
  • openEditModal()/saveEditToken() → GET /api/tokens then PATCH /api/tokens/{id}

5. Provider actions

Provider cards are driven by /api/providers. The modal writes provider defaults back with PATCH /api/providers/{provider_name}, and the live test button calls POST /api/providers/{provider_name}/test.

6. Logs tab

The logs page reads recent proxy traces from the persisted trace store:

  • loadLogs() → GET /api/logs
  • loadLogDetail(id) → GET /api/logs/{id}
  • The auto-refresh loop polls every 3 seconds while the Logs tab is active

The dashboard is DB-backed only. Tokens are managed directly in ~/.oauthrouter/tokens.db via the portal, the CLI, or the helper scripts.

Request flow on the server

Page routes

/portal and /help simply return static HTML files from src/oauthrouter/static/.

DB-backed dashboard APIs

Most portal endpoints are thin wrappers around TokenStore and configuration state. They validate JSON, read or mutate the SQLite DB, and return JSON shaped specifically for the browser renderer.

Provider tests

Provider probes are richer than plain CRUD. They can auto-pick tokens through TokenManager, respect cooldowns, retry on alternate tokens after 429, attempt auth recovery on 401/403, and update rate-limit snapshots for dashboard display.

Catch-all proxy

The final route is the real proxy entrypoint. It forwards requests through forward_request(), records trace attempts, keeps recent summaries in memory for the dashboard, and persists full traces under ~/.oauthrouter/logs/.

Routing Policy In Code

1. The URL prefix picks the provider lane

Provider routing is fixed by the incoming path. The router does not switch providers based on the request body model.

@app.api_route(
    "/{provider}/{path:path}",
    methods=["GET", "POST", "PUT", "DELETE", "PATCH", "OPTIONS", "HEAD"],
)
async def proxy_route(provider: str, path: str, request: Request) -> Response:
    response = await forward_request(
        request=request,
        provider=provider,
        path=path,
        ...
    )

Policy: /claude/* stays on Claude tokens and /openai/* stays on OpenAI tokens. There is no cross-provider failover and no model-driven provider switching.

2. The request body is forwarded unchanged

The router injects auth and forwards the original payload. It does not rewrite the model field.

upstream_url = _build_upstream_url(
    provider_config, path, request.url.query or ""
)
headers = _prepare_forwarded_headers(request)
headers = _inject_auth(headers, token, provider_config)
body = await request.body()

response = await _do_request(
    http_client=http_client,
    method=request.method,
    url=upstream_url,
    headers=headers,
    body=body,
    request_id=request_id,
)

Policy: the client’s requested model goes upstream exactly as sent. The router does token/account routing only; it does not do model fallback or model substitution.

3. Token order is priority-first, healthy-only

Selection is scoped to one provider and starts from DB priority order.

SELECT * FROM tokens
WHERE provider = ? AND status = 'healthy'
ORDER BY priority ASC, created_at ASC, id ASC

for candidate in healthy:
    if candidate.id in excluded:
        continue
    cooldown_until = self.get_rate_limit_cooldown(candidate.id)
    if cooldown_until is not None:
        continue
    candidates.append(candidate)

Policy: only healthy tokens are considered. Lower numeric priority wins. Tokens in in-memory cooldown are skipped. last_used_at is operational metadata only and is not part of routing order anymore.

4. Quota depletion means same-provider token failover

An upstream 429 is treated as token/account exhaustion, not as a model rewrite signal.

if response.status_code == 429:
    cooldown_until = _cooldown_until_from_headers(response.headers)
    token_manager.mark_token_rate_limited(token.id, retry_at=cooldown_until)

    token = await token_manager.pick_token(
        provider,
        exclude_token_ids={token.id},
    )
    response = await _do_request(...)

Policy: the router cools down the current token and retries the exact same request on the next token for the same provider. If no usable token remains, the request ends as 429. It does not switch models and it does not switch providers.

5. Auth failures try refresh before giving up

401/403 are treated differently from quota errors.

if response.status_code in (401, 403):
    next_token = await token_manager.handle_auth_failure(token, provider)

async def handle_auth_failure(self, token: Token, provider: str) -> Optional[Token]:
    refreshed = await self.refresh_token(token)
    if refreshed:
        return refreshed
    await self._store.mark_unhealthy(token.id)
    return await self.pick_token(provider, exclude_token_ids={token.id})

Policy: auth failures attempt token refresh first. If refresh fails, the token is marked unhealthy and the router moves to the next token for the same provider.

6. When every token is blocked, the router stops locally

The router can return an immediate local error before any upstream call if every healthy token is cooling down.

if not candidates:
    raise NoUsableTokensError(
        provider=provider,
        total_tokens=total_tokens,
        healthy_tokens=len(healthy),
        cooling_tokens=cooling_tokens,
        excluded_tokens=excluded_tokens,
        next_retry_at=next_retry_at,
    )

Policy: if the provider still has healthy tokens in the DB but all of them are cooling down, the router returns a local 429 with the next retry time instead of sending another doomed request upstream.

Other Important Snippets

Provider tests are health probes, not exact live-model tests

The dashboard test buttons use fixed probe models or the OpenAI usage endpoint.

PROBE_MODELS = {
    "claude": "claude-haiku-4-5-20251001",
    "openai": "gpt-4o-mini",
}

return (
    "POST",
    f"{provider_cfg.upstream.rstrip('/')}/v1/messages",
    headers,
    {
        "model": PROBE_MODELS["claude"],
        "max_tokens": 24,
        "messages": [{"role": "user", "content": "Reply with OK only."}],
    },
)

Policy: a passing provider test means the token/provider path is alive and rate-limit data can be collected. It does not prove that your exact live model still has quota.

The dashboard boot path is a thin data fan-out

The browser fetches state in parallel and then renders everything from those three payloads.

const [tokens, health, providers] = await Promise.all([
  api('/api/tokens'),
  api('/health'),
  api('/api/providers'),
]);

renderTokens(tokens);
renderStats(tokens, health);
renderProviders(providers);
setEndpointPorts();

Policy: the home screen is intentionally thin. Most behavior lives on the server; the browser mostly renders DB state and posts small mutations back to the API.

Rate-limit numbers are normalized for the UI

The server translates mixed provider formats into one shared snapshot shape for the portal.

def _parse_fractional_value(value: str, *, allow_overage: bool = False) -> Optional[float]:
    if numeric <= 1:
        return numeric
    if allow_overage and numeric < 10:
        return numeric
    if numeric <= 100:
        numeric /= 100.0
        return numeric

window["utilization"] = util
snapshot[f"{label}_reset"] = reset_iso

Policy: the dashboard always receives utilization in a normalized fraction form plus explicit reset timestamps. Anthropic overages like 1.01 are preserved as 101%, not misread as 1%.

Unknown API paths are kept out of the proxy lane

A small guard route prevents old or mistyped API URLs from being interpreted as provider names.

@app.api_route("/api/{path:path}", methods=[...])
async def api_not_found(path: str) -> JSONResponse:
    normalized = f"/api/{path}".rstrip("/") if path else "/api"
    return JSONResponse(
        {"error": f"API endpoint '{normalized}' not found"},
        status_code=404,
    )

Policy: bad API paths fail cleanly as API errors instead of falling through the provider proxy route as Unknown provider: api.
